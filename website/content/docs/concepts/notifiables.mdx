A **notifiable** is anything that can receive a notification — a user entity, a team, a DTO, a
plain object. There's no base class to extend and no decorator to add. You implement one
interface, and the recipient stays a normal object you own.

```ts
import type { Notifiable } from '@dudousxd/nestjs-notifications-core';

export class User implements Notifiable {
  constructor(
    public id: number,
    public email: string,
    public phone: string,
  ) {}

  routeNotificationFor(channel: string): unknown {
    switch (channel) {
      case 'mail':
        return this.email;
      case 'sms':
        return this.phone;
      default:
        return undefined;
    }
  }
}
```

## routeNotificationFor

This is the whole contract. Given a channel name, return the address for that channel — an email,
a phone number, a websocket room, a Slack webhook, or whatever the channel expects. The recipient
owns *where* a notification goes; the notification owns *what* it sends.

Return `undefined` for channels you don't route, and the channel will skip or fall back as
documented for that channel.

The second argument is the notification instance, so you can route differently per notification
when you need to:

```ts
routeNotificationFor(channel: string, notification: Notification): unknown {
  if (channel === 'mail' && notification instanceof SecurityAlert) {
    return this.securityEmail ?? this.email;
  }
  return this.email;
}
```

## toNotifiableRef — for the queue

Synchronous delivery passes your live object straight to the channels, so the basic contract is
all you need. The moment a notification is **queued**, though, it crosses a process boundary: the
worker that delivers it has no access to your in-memory `User`. It only has JSON.

So a queued notifiable must serialize to a stable reference the worker can reload:

```ts
import type { Notifiable, NotifiableRef } from '@dudousxd/nestjs-notifications-core';

export class User implements Notifiable {
  // …
  toNotifiableRef(): NotifiableRef {
    return { type: 'User', id: this.id };
  }
}
```

The worker rebuilds the recipient from that reference via the `resolveNotifiable` function you
give `NotificationsModule.forRoot()`. See [Async dispatch](/docs/concepts/async-dispatch) for the
full round trip.

<Callout type="info">
  Implement `toNotifiableRef` only if the notifiable might be queued. Sync-only apps never need it
  — and the library throws a clear error if you queue a notifiable that lacks it.
</Callout>

## On-demand notifiables

Sometimes there's no entity at all — you just want to email an address. You don't need a
notifiable for that; use [on-demand routing](/docs/recipes/on-demand):

```ts
await notifications.route('mail', 'ops@example.com').notify(new ServerDown());
```

Under the hood that builds an anonymous notifiable that routes the given channel to the given
value — the channels can't tell the difference.
